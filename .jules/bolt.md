## 2026-05-23 - [Optimize graph builder BFS sort]
**Learning:** Using `sort_by_cached_key` instead of `sort_by_key` for sorting adjacency lists during graph building offers a measurable performance improvement for large repositories (from 33ms to 31ms).
**Action:** Always consider `sort_by_cached_key` when sorting vectors by keys that require calculation or allocation (like `String` creation from an enum), especially on hot paths.
## 2026-05-23 - [Optimize HashSet Insertions with Pre-Checks]
**Learning:** In hot graph traversal loops (like BFS), blindly calling `visited.insert(value.clone())` where the value is an expensive type like `String` or `PathBuf` introduces significant performance overhead due to unnecessary heap allocations for already visited nodes.
**Action:** Consider pre-checking `!visited.contains(&value)` before an expensive `clone()` and `insert()` combination only when duplicate rates are high enough to offset the extra hash lookup. For mostly unique inputs, prefer a single `insert(value.clone())` probe.
## 2026-05-28 - [Optimize syn::parse_file with string pre-checks]
**Learning:** `syn::parse_file` is extremely expensive because it builds a full Abstract Syntax Tree (AST). In the context of linters or codebase analysis tools that look for specific attributes (like `#[allow(...)]` or `#[cfg(test)]`), blindly parsing every `.rs` file into an AST is a massive performance bottleneck.
**Action:** Always add a fast, O(N) string pre-check (e.g., `content.contains("allow")`) before attempting to parse the file into an AST. Since Rust attributes require these exact substrings in the unparsed code, the pre-check perfectly bounds the potential matches and allows the tool to safely and quickly skip irrelevant files, leading to ~3x performance improvements on the filesystem rules task.
## 2026-06-03 - [Optimize graph traversal HashSet/VecDeque insertions]
**Learning:** In string-heavy path traversal functions, tracking visited state and queues using `String` requires deep cloning on every node expansion. Switching from `&[String]` roots tracking `HashSet<String>` to tracking string slices `HashSet<&str>` using the original string references saves significant clone allocations. Combined with replacing map `.into_iter().flatten()` iteration with direct map `.get()` and simple inner loops (`if let Some`), this produces measurable traversal improvements (~6% faster execution).
**Action:** When tracking seen/visited identifiers in graphs built from Strings, always attempt to bind string slice lifetimes (`&'a str`) to the original collection or node lifetimes to prevent `.clone()` spam during iteration.
## 2026-06-16 - [Optimize nested iterator checks]
**Learning:** `file_allowed_by_roots_and_skip` in `crates/no-mistakes/src/codebase/rules/mod.rs` was redundantly evaluating `.iter().any(|rule_root| ... )` over the rule roots array multiple times, causing repeated `.starts_with` and `is_under_skipped_dir` evaluations.
**Action:** When performing `any()` checks that compute expensive values or need to test multiple conditions on the matching items, combine them into a single pass (`for ... in`) to short-circuit properly and avoid redundant function calls.
## 2026-06-25 - [Optimize BFS HashSet Insertions]
**Learning:** In Rust, when tracking visited nodes in a HashSet during hot loops (like graph traversals), unconditionally calling `visited.insert(value.clone())` with expensive types like `NodeId` or `PathBuf` causes unnecessary heap allocations for already visited elements.
**Action:** Use a pre-check `if !visited.contains(&value) { visited.insert(value.clone()); ... }` to prevent these allocations, resulting in measurable performance improvements in traversals.
