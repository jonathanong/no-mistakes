## 2026-05-23 - [Optimize graph builder BFS sort]
**Learning:** Using `sort_by_cached_key` instead of `sort_by_key` for sorting adjacency lists during graph building offers a measurable performance improvement for large repositories (from 33ms to 31ms).
**Action:** Always consider `sort_by_cached_key` when sorting vectors by keys that require calculation or allocation (like `String` creation from an enum), especially on hot paths.
## 2026-05-23 - [Optimize HashSet Insertions with Pre-Checks]
**Learning:** In hot graph traversal loops (like BFS), blindly calling `visited.insert(value.clone())` where the value is an expensive type like `String` or `PathBuf` introduces significant performance overhead due to unnecessary heap allocations for already visited nodes.
**Action:** Consider pre-checking `!visited.contains(&value)` before an expensive `clone()` and `insert()` combination only when duplicate rates are high enough to offset the extra hash lookup. For mostly unique inputs, prefer a single `insert(value.clone())` probe.
## 2026-05-28 - [Optimize syn::parse_file with string pre-checks]
**Learning:** `syn::parse_file` is extremely expensive because it builds a full Abstract Syntax Tree (AST). In the context of linters or codebase analysis tools that look for specific attributes (like `#[allow(...)]` or `#[cfg(test)]`), blindly parsing every `.rs` file into an AST is a massive performance bottleneck.
**Action:** Always add a fast, O(N) string pre-check (e.g., `content.contains("allow")`) before attempting to parse the file into an AST. Since Rust attributes require these exact substrings in the unparsed code, the pre-check perfectly bounds the potential matches and allows the tool to safely and quickly skip irrelevant files, leading to ~3x performance improvements on the filesystem rules task.
